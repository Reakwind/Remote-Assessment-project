The "Test Contract"

To keep things flexible, every test we build will follow the same rules (a "Contract"):

  • Input: What specific instructions or parameters does this test need? (e.g. "Show 5 items for 10 seconds")
  • Internal State: The test handles its own timer and user clicks.
  • Output: When finished, it must return a standardized packet: { testId: "memory-1", score: 85, timeTaken: 45, rawData: [...] }

Accessibility Safety Net

  • Audio Fallback: The engine will have a built-in "Read Aloud" feature for instructions.
  • No-Trap Navigation: Navigation buttons (Next/Back) are managed by the Engine, not the individual tests, so the user can never get "stuck" inside a broken test.
