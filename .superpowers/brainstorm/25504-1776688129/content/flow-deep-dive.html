Deep Dive: Managing the Assessment Flow

Comparing approaches for flexibility, reliability, and ease of change.

A

Hard-Coded Flow (The "Fixed Script")

Analogy: A printed book. To change the order of chapters, you have to reprint the whole book.

  • Fastest to build initially
  • Zero hidden complexity
  • Hard to reorder tests
  • Adding a test requires changing core code
B

Config-Driven (The "Playlist")

Analogy: A Spotify playlist. You can drag and drop songs (tests) without changing how the music player works.

  • Easy to Change: Swap tests by editing one list.
  • Resilient: Easy to save "Current Track" if the user's internet drops.
  • Requires building a "Player" engine first.
C

State Machine (The "Air Traffic Control")

Analogy: A complex flowchart. If X happens, go to Y; if user is tired, skip to Z.

  • Handles complex logic perfectly
  • Mathematically impossible to get in a "broken state"
  • Highest learning curve
  • Overkill for simple sequential batteries

My Recommendation: Option B (Config-Driven)

For your "Developer-Led" requirement, Option B is the "Sweet Spot".

  1. Maintenance: When you want to add a new "Memory Test", you just add it to the library and update the battery.json file. You don't touch the navigation code.
  2. Remote Reliability: Since the flow is just a list of steps, it's very easy to save the patient's progress to Supabase after every step. If their browser crashes, they come back and the "Player" just looks at the list and resumes at step #3.
  3. Flexibility: It allows you to have different batteries (e.g., "Short Version" vs "Full Version") just by using different JSON files.
